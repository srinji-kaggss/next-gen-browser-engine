# 2026-07-01 — Phase 2 Spec Creation

## Date

2026-07-01

## Hypothesis

A JS-to-Braid transpiler + HTTP network stack can be built as parallel crates, enabling real page fetching and JavaScript verification.

## Design

Two new crates in the browser-engine workspace:

- **be-transpiler** — JS→Braid transpiler using SWC for parsing/AST. Exposes a strategy registry pattern (not if-chains) for extensible JS feature support.
- **be-net** — HTTP network stack using reqwest. Handles fetching, redirects, cookie jars, and response parsing.

Three new API endpoints:

- `GET /fetch?url=` — Fetch a URL and return raw response
- `POST /transpile` — Transpile JS source to Braid IR
- `POST /load?url=` — Fetch + transpile in one call

## Raw Data

- MVP 1 is complete: 12 crates, 10 tests passing
- SWC dep: `swc = { version = "0.1", features = ["typescript", "verify"] }`
- reqwest dep: `reqwest = { version = "0.13", features = ["rustls-tls", "cookies", "gzip"] }`
- Strategy pattern sketch: `trait TranspileFeature { fn transpile(&self, node: &EsNode) -> Option<BraidIR>; }` with a `FeatureRegistry` that dispatches by AST node type

## Observation

- **Strategy registry > if-chains.** Adding JS features (destructuring, async/await, etc.) should require registering a new strategy, not modifying a central dispatcher.
- **Capability inference needs escalation records.** When the transpiler encounters an unsupported JS feature, it should record the feature and its AST node type so capability gaps are trackable.
- **be-net is simpler than be-transpiler.** Start with be-net to validate crate structure and test patterns before tackling transpilation.
